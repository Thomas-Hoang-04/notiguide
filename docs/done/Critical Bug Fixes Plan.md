### Task 1: Fix Store Creation NPE

**Context:** `StoreService.createStore()` crashes with NPE because Spring Data R2DBC's `save()` returns the `@Id` column (`id`) but not the DB-generated `publicId` column. The `toDto()` call uses `publicId!!` which throws NPE. The fix is a re-fetch after save to get the full row including `publicId`.

**Files:**
- Modify: `backend/src/main/kotlin/com/thomas/notiguide/domain/store/service/StoreService.kt:111`

- [ ] **Step 1: Replace the return line in createStore()**

In `StoreService.kt`, replace line 111:

```kotlin
        return saved.toDto()
```

with:

```kotlin
        val complete = storeRepository.findById(saved.id!!)
            ?: throw IllegalStateException("Store not found after save")
        return complete.toDto()
```

The full `createStore()` method after the change (lines 85-114):

```kotlin
    @Transactional
    suspend fun createStore(request: CreateStoreRequest): StoreDto {
        require(request.name.isNotBlank()) { "Store name must not be blank" }
        validateNoShowAction(request.noShowAction)

        val store = Store(
            name = request.name,
            address = request.address?.takeIf { it.isNotBlank() },
            allowJumpCall = request.allowJumpCall,
            allowNoShow = request.allowNoShow
        )
        val saved = storeRepository.save(store)

        storeSettingsRepository.save(
            StoreSettings(
                storeId = saved.id!!,
                maxQueueSize = request.maxQueueSize,
                gracePeriodSec = request.gracePeriodSec,
                noShowAction = request.noShowAction,
                maxRequeues = request.maxRequeues,
                requeueOffset = request.requeueOffset,
                alertThreshold = request.alertThreshold
            ).markNew()
        )

        serviceTypeRepository.save(ServiceType(storeId = saved.id, name = "General", prefix = "A"))

        val complete = storeRepository.findById(saved.id!!)
            ?: throw IllegalStateException("Store not found after save")
        return complete.toDto()
    }
```

**Why this works:**
- `saved.id!!` is safe — `id` is the `@Id` column and IS populated by R2DBC's RETURNING
- `findById()` returns the full row including `publicId` generated by PostgreSQL's `generate_store_public_id()` DEFAULT
- Within the same `@Transactional`, PostgreSQL guarantees read-your-own-writes — no delay needed
- `storeSettingsRepository.save()` and `serviceTypeRepository.save()` are unchanged — they only use `saved.id!!` which works correctly

### Task 2: Fix nRF24 GPIO Pin Collision

**Context:** The receiver's `sdkconfig` assigns both the nRF24 IRQ pin and the vibrator to GPIO 1. This was a typo — the IRQ was intended for GPIO 10. The collision causes the nRF24 chip to not respond to SPI commands after provisioning, returning 0xFF for all register reads.

**Files:**
- Modify: `receiver-esp32/sdkconfig:842`

- [ ] **Step 1: Change nRF24 IRQ pin from GPIO 1 to GPIO 10**

In `receiver-esp32/sdkconfig`, change line 842:

```
CONFIG_RECEIVER_NRF24_IRQ=1
```

to:

```
CONFIG_RECEIVER_NRF24_IRQ=10
```

**Pin allocation after fix:**

| GPIO | Function |
|------|----------|
| 1 | Vibrator |
| 2 | nRF24 MISO |
| 3 | nRF24 CE |
| 5 | nRF24 CS |
| 6 | nRF24 SCK |
| 7 | nRF24 MOSI |
| 10 | nRF24 IRQ |

GPIO 10 is confirmed free — not used by any other peripheral in sdkconfig or source code.

### Task 3: Add Defensive Vibrator Stop at Boot

**Context:** Even with the GPIO fix, the vibrator should be explicitly set to OFF after initialization as defense-in-depth. This prevents any future state leak from leaving the vibrator buzzing on boot.

**Files:**
- Modify: `receiver-esp32/main/main.c:128-129`

- [ ] **Step 1: Add rf_trigger_stop_output() after rf_trigger_init()**

In `receiver-esp32/main/main.c`, the current lines 128-129 are:

```c
    ESP_ERROR_CHECK(rf_trigger_init((gpio_num_t)CONFIG_RECEIVER_VIBRATOR_GPIO));
    restore_trigger_state();
```

Change to:

```c
    ESP_ERROR_CHECK(rf_trigger_init((gpio_num_t)CONFIG_RECEIVER_VIBRATOR_GPIO));
    rf_trigger_stop_output();
    restore_trigger_state();
```

**Why this is safe:**
- `rf_trigger_init()` (at `rf_trigger.c:46-61`) creates the mutex, calls `vibrator_init()`, and sets `s_trigger.initialized = true`
- `rf_trigger_stop_output()` (at `rf_trigger.c:126-132`) checks `s_trigger.initialized` — this is now `true`, so the guard passes
- It calls `vibrator_set_pulsing(&s_vibrator, false)` which acquires the lock, sets `vibrator_pulsing = false`, and forces GPIO low if active
- `restore_trigger_state()` only sets RF code data (bytes/bits/version) — it does NOT start the vibrator
- No `ESP_ERROR_CHECK` needed — matches the existing pattern at `main.c:84` where `rf_trigger_stop_output()` is called without error checking

### Task 4: Fix Receiver Main Task Stack Size

**Context:** The receiver's main task stack (6144 bytes) is 2 KB smaller than the transmitter's (8192 bytes). The receiver's `app_main()` performs heavy operations (HTTP server, WiFi, MQTT, cJSON, NVS) that overflow the smaller stack on the single-core ESP32-C3, causing a hardware stack guard panic reboot.

**Files:**
- Modify: `receiver-esp32/sdkconfig:1470,2907`

- [ ] **Step 1: Increase CONFIG_ESP_MAIN_TASK_STACK_SIZE**

In `receiver-esp32/sdkconfig`, change line 1470:

```
CONFIG_ESP_MAIN_TASK_STACK_SIZE=6144
```

to:

```
CONFIG_ESP_MAIN_TASK_STACK_SIZE=8192
```

- [ ] **Step 2: Update the legacy alias**

In the same file, change line 2907:

```
CONFIG_MAIN_TASK_STACK_SIZE=6144
```

to:

```
CONFIG_MAIN_TASK_STACK_SIZE=8192
```

Both values must match. `CONFIG_MAIN_TASK_STACK_SIZE` is a deprecated alias for `CONFIG_ESP_MAIN_TASK_STACK_SIZE` — ESP-IDF sdkconfig keeps both in sync.
